## Decisions & deviations (append-only log)
- Preview strategy: live iframe of deployed demo for web apps; recorded media/snippets for cli/service/library/notebook/media. No arbitrary-repo builds.
- Hosting: Vercel for frontend; AWS enters only with the FastAPI service (Phase 3).
- Backend: Python/FastAPI, introduced in Phase 2→3 (metrics can start in Next.js route handlers; FastAPI owns AI + heavier analytics).
- Animation: used `motion` (the maintained successor package) in place of `framer-motion`, per current Context7 docs — same API (`useScroll`, `whileInView`), imported from `motion/react`.
- Content: Velite (not Content Collections) generates a typed content layer from MDX, validated against `projectSchema` at build time; `strict: true` fails the build on invalid frontmatter/payloads (covered by an integration test in `test/fixtures/velite-invalid-content`).
- Schema hardening: `projectSchema` uses `z.strictObject()` throughout and a `previewType`-discriminated union so each preview variant only carries its own fields (e.g. a `cli` project cannot also hold `demoUrl`); `webapp.demoUrl` is required, matching the `<Preview>` exhaustiveness check in Slice 5.
- Dark mode: `next-themes` with `attribute="class"`, matching shadcn's existing `.dark` CSS-variable block and `@custom-variant dark` in `globals.css` — no extra Tailwind config needed.
- Pagination: client-side, `PAGE_SIZE = 6`, revealed via an `IntersectionObserver` sentinel at the end of the mounted list (not a real network fetch). The 4 shipped sample projects don't exceed one page, so this is exercised by a dedicated unit test with synthetic data rather than visibly in the current content set.
- Mobile single rail: implemented via responsive `grid-template-columns` + explicit `col-start`/`row-start` per breakpoint on the existing alternating-side markup, not by duplicating or hiding cards — every project card still mounts exactly once regardless of viewport.
- Phase 2 infra: Neon branching for test/dev/prod DBs; Vercel Blob for capture media (S3 in Phase 3, behind storagePut()); Upstash Redis for /api/events rate-limiting. All serverless, no AWS.
- Slice 1: lazy Drizzle client init via `Proxy` (`src/db/client.ts`, `src/db/tx-client.ts`) so importing `@/db/queries` never throws at module-load time when `DATABASE_URL` is unset — only an actual query does. This lets `queries.integration.test.ts` load and `describe.skipIf` cleanly instead of crashing the whole test file when run without a DB configured.
- Slice 1 gotcha (fixed): `drizzle-kit`'s own `dotenv` reload inside `drizzle.config.ts` refills any `.env.local`-defined var that isn't already present in `process.env` — so `scripts/db-migrate-test.mjs` must set both `DATABASE_URL` and `DATABASE_URL_UNPOOLED` to the test branch's URL (not just delete the latter), or the reload silently redirects migrations back to the `dev` branch while still reporting success.
- Slice 2: ISR via the page-level `export const revalidate` route-segment config (Context7-confirmed this works for any Server Component, not just `fetch`-based data) rather than `unstable_cache` — simpler for a single top-level read, and this route has no per-request dynamic APIs so it's still fully static + revalidated. On-demand cache-busting (`revalidateTag`/`revalidatePath` fired from an admin write) is deferred to Slice 4, when writes exist.
- Slice 2: since ISR bakes the page at build time from whatever's in the DB, the Test Gate must run `pnpm db:seed` (against the Neon `dev` branch) *before* `pnpm build`, or the prerendered page reflects an empty/stale table. `predb:seed` runs `velite` first so the seed always reads fresh MDX-derived content.
- Slice 2: `upsertProject`'s `updatedAt` is set via `sql\`now()\`` (the Postgres server's own clock) rather than a Node.js `new Date()` — mixing the two clock sources caused a flaky integration-test assertion (`inserted.updatedAt < updated.updatedAt`) under real client/server clock skew against Neon.
- Slice 3: authN (GitHub OAuth) and authZ (`ADMIN_EMAILS` allowlist) are deliberately decoupled — any GitHub user may *authenticate* (the `signIn` callback only asserts `provider === "github"`); *authorization* is a separate pure `isAdmin(email, adminEmails)` check. A non-allowlisted user can sign in but reaches nothing protected. Keeps the allowlist reusable by both the middleware gate and the resource-layer guard without dragging OAuth/session types into it.
- Slice 3: middleware is treated as an OPTIMISTIC UX gate ONLY, never the security boundary (CVE-2025-29927 makes it bypassable). The AUTHORITATIVE check is re-run server-side at every protected resource via `require-admin.ts`, which re-derives admin-ness from the *current* `ADMIN_EMAILS` against the session's verified email rather than trusting a stale `isAdmin` flag baked into the JWT — so revoking an admin takes effect immediately, and a bypassed middleware still fails closed. Covered by a handler-level test (`route.test.ts`) that denies without any middleware in the loop.
- Slice 3: matcher uses a POSITIVE allowlist (`["/admin/:path*","/api/admin/:path*"]`) instead of a negative lookahead. This guarantees `/` + public assets + `/api/auth/**` never enter the middleware, so the public timeline stays fully static/ISR and the auth flow isn't self-gated — verified by the build output (`/` still `○ Static`) and an E2E asserting `/` is untouched.
- Slice 3: GitHub can return a `null` email for private-email accounts, so the provider requests `read:user user:email` and the `jwt` callback fetches the primary VERIFIED email from GitHub `/user/emails` on first sign-in (access token only present then; no per-request network call on the hot path), stamping it onto the token/session. The allowlist compares against that verified email — otherwise the developer would lock themselves out. Manual lock-out check (log in locally, confirm the match) is the first acceptance step before trusting denials.
- Slice 3: `trustHost: true` set in the shared `auth.config.ts` (a plain, edge-safe boolean). Auth.js auto-detects a trusted host on Vercel (prod) but NOT under local `next dev`/`next start`; without it, BOTH the middleware instance and the root instance throw `UntrustedHost`, which silently masqueraded as "denied" in the first E2E run and would have broken the developer's local sign-in verification. No new env var introduced (chose the config boolean over `AUTH_TRUST_HOST` to avoid provisioning a var Phase 2 didn't already have). Safe for a single-domain deployment where the platform sets the host header.
- Slice 3: authZ tests inject the authenticated principal (mock `auth()` / drive the `authorized` callback directly) rather than driving real GitHub OAuth in E2E — deterministic, no network, and exercises the exact allowlist matrix (incl. the critical authenticated-but-NOT-allowlisted denial) at both the resource layer and the optimistic gate. E2E only covers the no-session real-middleware path (redirect + write denial) and that `/` is untouched.
- Slice 4: ingested projects render as **metadata-only timeline nodes** this slice. `previewType` + `demoUrl` are collected and persisted as columns, but NOT rendered — rendering the discriminated preview union is the later frontend preview slice (already in Slice 4's out-of-scope guards). To make this safe with the existing strict read path: `preview` jsonb was made nullable, a preview-less `projectMetadataSchema` + `projectRowToTimelineNode` mapper were added, and the public `TimelineProject` type made `preview` optional. The mapper still re-validates every row (full `projectSchema` when a preview jsonb exists, `projectMetadataSchema` otherwise), so a corrupt ingested row throws in the mapper (caught by the Slice 2 page-level try/catch) rather than reaching the timeline — and `ProjectCard` renders the preview surface + Demo link only when `preview` is present. Seeded MDX projects (which keep their full `preview` jsonb) are byte-for-byte unchanged.
- Slice 4: reused existing columns instead of adding redundant ones — the spec's "published" boolean is the existing `status` enum (`draft|published`), "description" is `summary`, "canonical repoUrl" is the existing `github_url` (rebuilt canonically from owner/repo, never the pasted string), and the timeline-ordering date is the existing `start_date` (defaulted from GitHub `created_at`, admin-editable). Only genuinely-new fields (github identity/metadata, `demo_url`, `preview_type`) were added. The `UNIQUE(github_owner, github_repo)` pair is nullable, and Postgres treats NULLs as distinct, so the 4 pre-existing seed rows (owner/repo both NULL) don't collide — only a second row with the same non-null owner+repo violates it, which is exactly the duplicate-ingestion rejection.
- Slice 4: SSRF guard is structural, not a URL allowlist bolt-on — `parseGithubRepoUrl` only extracts `{owner, repo}` and `fetchRepoMetadata` fetches ONLY `https://api.github.com/repos/{owner}/{repo}`. The pasted URL is never fetched, and a `demoUrl` is validated (https) and STORED but never fetched. Non-`github.com` hosts (including look-alikes like `github.com.evil.com` and internal IPs like `169.254.169.254`) are rejected at parse time.
- Slice 4: `GITHUB_TOKEN` is read as an OPTIONAL env var only (Bearer -> 5000/hr; absent -> unauthenticated 60/hr) — not created, not required, and no other new env var introduced (matches the HANDOFF "external provisioning complete" rule). GitHub requires a `User-Agent` header on API requests, so one is always sent.
- Slice 4: duplicate detection walks the error `.cause` chain for Postgres SQLSTATE `23505` — Drizzle wraps the neon-serverless driver error in a `DrizzleQueryError`, so the pg `.code` is on `.cause`, not the top-level error (a top-level-only check silently failed the first integration run, surfacing the raw "Failed query" error instead of `DuplicateProjectError`).
- Slice 4: `setPublished` fires `revalidatePath("/")` on every publish/unpublish — this is the on-demand ISR cache-busting Slice 2 explicitly deferred "to Slice 4, when writes exist," so a newly published project shows on the public timeline without waiting for the 1h revalidation window or a redeploy.
- Slice 5: the anonymous event beacon records an EPHEMERAL, in-memory-only, per-page-load `crypto.randomUUID()` session id (ratified amendment) to satisfy the Slice 1 NOT-NULL `events.session_id` column WITHOUT a migration or a sentinel constant. It is a session-scoped correlation id, NOT a visitor id: never persisted (no cookie/`localStorage`/`sessionStorage`), never IP/fingerprint-derived, dies on refresh, never joined to identity, and surfaced only as "sessions" (never "unique visitors"). The strict payload is therefore `{projectId, type, sessionId}` — the kickoff's "projectId + type only" plus this ratified id; a client `ts` (or any other extra key) is still rejected, and the event timestamp is always the DB's `now()`.
- Slice 5: "project expanded" reuses the EXISTING `hover` enum value rather than adding one (ratified: no enum values added, no enum migration). The event fires ONLY on the deliberate "Preview" button click (a real user action), never on hover-intent (noisy, no touch equivalent, not intent) — implemented by splitting `WebappPreview.handleExpandClick` (button `onClick` → `startLoading` + `onExpand`) from the hover path (`startLoading` only). The legacy `hover` label is an internal DB value; the dashboard surfaces it as "Preview opened" so the misnomer never reaches the UI. Only `webapp` previews have a click-to-open surface, so only they can emit this event; other preview types render inline with nothing to expand.
- Slice 5: event WRITES go through the `neon-serverless` Pool writer (`recordEvent` → `txDb`) per the Slice 1 driver split, even though a single-row insert isn't transactional — kept for the "reads = http, writes = Pool" invariant. This surfaced a production-only bug: under `next start`, Next bundled `ws` (the Pool's WebSocket transport) and mangled its buffer-mask fallback (`TypeError: b.mask is not a function`), 500-ing every event write. This was latent since Slice 4 (whose Pool writes are admin-only and never hit by e2e); the public beacon is the first Pool write on a hot, e2e-exercised path.
- Slice 5: rate limiting is Upstash Redis only (never Postgres), keyed on the client IP used TRANSIENTLY as the limiter key and never stored. `checkEventRateLimit` FAILS OPEN on a limiter error (logs, then allows) — the beacon is fire-and-forget and never affects the UI, so losing analytics to an Upstash blip is worse than briefly relaxing the cap. The limiter is lazily constructed (like the DB clients) so importing the module never requires the env to be set.
- Slice 5: the public `/api/events` route is deliberately kept OUT of the middleware matcher (still the positive allowlist `["/admin/:path*","/api/admin/:path*"]`) so it stays unauthenticated and `/` stays static — consistent with the Slice 3 reasoning. authZ on the dashboard is the authoritative `requireAdmin()` as the first statement, matching the north-star resource-layer rule.
